3   /**
4    * The IMC and TNCC use this IMCConnection object to refer 
5    * to the network connection when delivering messages 
6    * and performing other operations relevant to the 
7    * network connection. This helps ensure that IMC 
8    * messages are sent to the right TNCS and helps the 
9    * IMC match up messages from IMVs with any state the 
10   * IMC may be maintaining from earlier parts of that 
11   * IMC-IMV conversation (even extending across multiple 
12   * Integrity Check Handshakes in a single network 
13   * connection).
14   * <p>
15   * The TNCC MUST create a new <code>IMCConnection</code> object for each
16   * combination of an IMC and a connection. <code>IMCConnection</code>
17   * objects MUST NOT be shared between multiple IMCs.
18   */

58  public interface IMCConnection {
59  
60      /**
61       * Handshake retry reason when IMC has completed remediation.
62       */
63  
64      public static final long TNC_RETRY_REASON_IMC_REMEDIATION_COMPLETE = 0;
65      
66      /**
67       * IMC has detected a serious event and recommends 
68       * handshake retry even if network connectivity must be interrupted.
69       */
70  
71      public static final long TNC_RETRY_REASON_IMC_SERIOUS_EVENT = 1;
72      
73      /**
74       * IMC has detected an event that it would like to communicate to 
75       * the IMV. It requests handshake retry but not if network 
76       * connectivity must be interrupted.
77       */
78  
79      public static final long TNC_RETRY_REASON_IMC_INFORMATIONAL_EVENT = 2;
80      
81      /**
82       * IMC wishes to conduct a periodic recheck. It recommends 
83       * handshake retry but not if network connectivity must be 
84       * interrupted.
85       */
86  
87      public static final long TNC_RETRY_REASON_IMC_PERIODIC = 3;
88      
89      /**
90       * An IMC calls this method to give a message to the TNCC for 
91       * delivery. The message is contained in the buffer referenced by 
92       * the message parameter.  The message parameter may 
93       * be null which represent an empty message. The type 
94       * of the message is indicated by the messageType parameter. 
95       * <p>
96       * All IMC Connections MUST implement this method. The IMC 
97       * Connection MUST NOT ever modify the buffer contents and MUST NOT 
98       * access the buffer after the TNCC sendMessage() method has 
99       * returned. The TNC Client will typically copy the message out of 
100      * the buffer, queue it up for delivery, and return from this method.
101      * <p>
102      * The IMC MUST NOT call this method unless it has received a 
103      * call to the IMC's beginHandshake() method, the IMC's 
104      * receiveMessage() method, or the IMC's batchEnding() method
105      * for this connection and the IMC has not yet returned from that 
106      * method. If the IMC violates this prohibition, the TNCC SHOULD 
107      * throw a TNC_RESULT_ILLEGAL_OPERATION exception. If an IMC 
108      * really wants to communicate with an IMV at another time, it 
109      * should call the TNCC requestHandshakeRetry() method.
110      * <p>
111      * Note that a TNCC or TNCS MAY cut off IMC-IMV communications at 
112      * any time for any reason, including limited support for long 
113      * conversations in underlying protocols, user or administrator 
114      * intervention, or policy. If this happens, the TNCC will throw a 
115      * TNC_RESULT_ILLEGAL_OPERATION. If the TNCC supports limiting the
116 	 * message size or number of round trips, the TNCC MUST throw a 
117 	 * TNCException with result code TNC_RESULT_EXCEEDED_MAX_MESSAGE_SIZE
118 	 * or TNC_RESULT_EXCEEDED_MAX_ROUND_TRIPS respectively if the 
119 	 * limits are exceeded.  An IMC can get the Maximum Message Size 
120 	 * and Maximum Number of Round Trips by using the related Attribute
121 	 * ID with the getAttribute method on the IMCConnection object.  
122 	 * The IMC SHOULD adapt its behavior to accommodate these 
123 	 * limitations if available.
124      * <p>
125      * The TNC Client MUST support any message type. However, the 
126      * IMC MUST NOT specify a message type whose vendor ID is 0xffffff 
127      * or whose subtype is 0xff. These values are reserved for use as 
128      * wild cards.  If the IMC violates 
129      * this prohibition, the TNCC SHOULD throw a
130      * TNC_RESULT_INVALID_PARAMETER exception. If the IMCConnection 
131 	 * supports limiting the message size or number of round trips, 
132 	 * the IMCConnection MUST return TNC_RESULT_EXCEEDED_MAX_MESSAGE_SIZE
133 	 * exception or TNC_RESULT_EXCEEDED_MAX_ROUND_TRIPS exception 
134 	 * respectively if the limits are exceeded.  An IMC can get the 
135 	 * Maximum Message Size and Maximum Number of Round Trips by using
136 	 * the related TNC_ATTRIBUTEID within the 
137 	 * AttributeSupport.GetAttribute method.  The IMC SHOULD adapt its
138 	 * behavior to accommodate these limitations if available.
139      * <p>
140      * @param messageType the type of message to be delivered
141      * @param message the message to be delivered 
142  
143      * @exception TNCException if a TNC error occurs
144      */
145 
146     public void sendMessage(long messageType, byte[] message) throws TNCException;
147 
148    /**
149     * An IMC calls this method to ask a IMC Connection to retry an
150     * Integrity Check Handshake. The IMC MUST pass one of 
151     * the handshake retry reasons listed in IF-IMC 
152     * Abstract API as the reason parameter.
153     * <p>
154     * TNCCs MAY check the parameters to make sure they are valid and 
155 	* throw an exception if not, but TNCCs are not required to make 
156 	* these checks. The reason parameter explains why the IMC is 
157 	* requesting a handshake retry. The TNCC MAY use this in deciding 
158 	* whether to attempt the handshake retry. As noted in the Abstract
159 	* API, TNCCs are not required to honor IMC requests for handshake 
160 	* retry (especially since handshake retry may not be possible or 
161 	* may interrupt network connectivity). An IMC MAY call this method
162 	* at any time, even if an Integrity Check Handshake is currently 
163 	* underway. This is useful if the IMC suddenly gets important 
164 	* information but has already finished its dialog with the IMV, 
165 	* for instance. As always, the TNCC is not required to honor the 
166 	* request for handshake retry.
167     * <p>
168     * If the TNCC cannot attempt the handshake retry, the IMCConnection
169 	* SHOULD throw a TNCException with result code 
170 	* TNC_RESULT_CANT_RETRY. If the TNCC could attempt to retry the 
171 	* handshake but chooses not to, the IMCConnection SHOULD throw a 
172 	* TNCException with result code TNC_RESULT_WONT_RETRY. The IMC MAY
173 	* use this information in displaying diagnostic and progress 
174 	* messages. 
175     * <p>
176     * @param reason the reason for the handshake request
177     *
178     * @exception TNCException if a TNC error occurs
179     */
180     
181     public void requestHandshakeRetry(long reason) throws TNCException;
182     
183 }
