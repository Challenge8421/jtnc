52     /**
53      * An IMC calls this method to give a message to the TNCC for 
54  	* delivery. This method provides several features that sendMessage
55  	* does not: longer 32 bit vendor ID and message subtype fields, 
56  	* a messageFlags field, and the ability to specify a source IMC ID
57  	* and a desired or exclusive destination IMV ID.
58      * <p>
59      * The message is contained in the buffer referenced by message. 
60  	* This parameter may be null to represent an empty message. The 
61  	* type of the message is indicated by the messageVendorID and 
62  	* messageSubtype parameters. TNCCs MAY check these values to make 
63  	* sure they are valid and throw an exception if not, but TNCCs are
64  	* not required to make these checks.
65      * <p>
66      * The messageFlags parameter supports up to 32 flags that may be 
67  	* set pertaining to the message. At this time, only one flag has 
68  	* been defined. The value 0x80000000 is known as the 
69  	* TNC_MESSAGE_FLAGS_EXCLUSIVE flag ("the EXCL flag", for short). 
70  	* If this bit is set, the sending IMC is requesting that the 
71  	* message only be delivered to the IMV designated by the 
72  	* destinationIMVID parameter. If the bit is cleared, the message 
73  	* is to be delivered to any IMV that has indicated an interest in 
74  	* the message type. Other flags may be defined in future versions 
75  	* of this specification. Until that time, IMCs MUST leave all 
76  	* these flags cleared. Some connections and/or TNCCs may not 
77  	* support the EXCL flag. To determine whether a particular TNCC 
78  	* and connection supports this flag, the IMC should get the Has 
79  	* Exclusive attribute for that connection. If the IMC sets a flag
80  	* that is not supported by the TNCC or the connection, the TNCC 
81  	* SHOULD throw a TNCException with a TNC_RESULT_INVALID_PARAMETER
82  	* result.
83      * <p>
84      * The IMC MUST set the sourceIMCID parameter to either the primary
85  	* IMC ID queried by IMC using the TNC_AttributeID_PRIMARY_IMC_ID 
86  	* attribute or an additional IMC ID reserved when IMC requests the
87  	* TNCC to do so by calling reserveAdditionalIMCID. This IMC ID 
88  	* will be used as the sourceIMCID for protocols that support 
89  	* sending this field along with the message.
90      * <p>
91      * If the EXCL flag is set, the IMC MUST set the destinationIMVID 
92  	* parameter to indicate which IMC should receive the message being
93  	* sent. The IMC should place into this parameter a value which it
94  	* has obtained using the sourceIMCID parameter for messages 
95  	* previously delivered to the IMC on this connection. The IMC MAY 
96  	* set the destinationIMVID parameter to a specific IMV ID even 
97  	* when the EXCL flag is not set. This indicates that the message 
98  	* being sent is in response to a message received from the IMV 
99  	* indicated in the destinationIMVID parameter. If the IMC does not
100 	* know the IMV ID of the recipient IMV, the IMC MUST use the 
101 	* TNC_IMVID_ANY wild card for the destinationIMVID parameter and 
102 	* MUST NOT set the EXCL flag. In such case, the message is 
103 	* delivered to all IMVs that have expressed interest in receiving 
104 	* such messages. This may happen when the TNCC starts the integrity
105 	* handshake and the IMC does not know the IMV IDs of the recipient
106 	* IMVs.
107     * <p>
108     * TNCCs are not required to implement this method but any TNCC 
109 	* that implements the IMCConnectionLong interface MUST implement 
110 	* this method. Since this method was not included in IF-IMC 1.0 
111 	* through 1.2, many TNCCs do not implement it. IMCs MUST work 
112 	* properly if a TNCC does not implement this method. The IMC is 
113 	* never required to call this method. The TNCC MUST work with IMCs
114 	* that don't call this method. The IMC also SHOULD check the Has 
115 	* Long attribute for a given connection to determine whether this 
116 	* connection supports long types before calling this method for 
117 	* that connection. If the connection does not support long types,
118 	* the IMC MAY call this method (if implemented) but MUST NOT use a
119 	* messageVendorID greater than 0xffffff or a messageSubtype 
120 	* greater than 0xff.  If the IMC does so, the TNCC SHOULD throw a 
121 	* TNC_RESULT_NO_LONG_TYPES exception.
122     * <p>
123     * The TNC Client MUST NOT ever modify the buffer contents and MUST
124 	* NOT access the buffer after sendMessageLong has returned. The 
125 	* TNC Client will typically copy the message out of the buffer, 
126 	* queue it up for delivery, and return from this method.
127     * <p>
128     * The IMC MUST NOT call this method unless it has received a call 
129 	* to beginHandshake, receiveMessage, receiveMessageSOH, 
130 	* receiveMessageLong, or batchEnding for this connection and the 
131 	* IMC has not yet returned from that method. If the IMC violates 
132 	* this prohibition, the TNCC SHOULD throw 
133 	* TNC_RESULT_ILLEGAL_OPERATION exception. If an IMC really wants 
134 	* to communicate with an IMV at another time, it should call 
135 	* requestHandshakeRetry.
136     * <p>
137     * Note that a TNCC or TNCS MAY cut off IMC-IMV communications at 
138 	* any time for any reason, including limited support for long 
139 	* conversations in underlying protocols, user or administrator 
140 	* intervention, or policy. If this happens, the TNCC will throw 
141 	* TNC_RESULT_ILLEGAL_OPERATION exception from sendMessageLong.  If
142 	* the TNCC supports limiting the message size or number of round 
143 	* trips, the TNCC MUST throw TNC_RESULT_EXCEEDED_MAX_MESSAGE_SIZE 
144 	* or TNC_RESULT_EXCEEDED_MAX_ROUND_TRIPS exception respectively if
145 	* the limits are exceeded.  An IMC can get the Maximum Message 
146 	* Size and Maximum Number of Round Trips by using the related 
147 	* Attribute ID within the getAttribute method.  The IMC SHOULD 
148 	* adapt its behavior to accommodate these limitations if available.
149     * <p>
150     * The TNC Client MUST support any message type. However, the IMC 
151 	* MUST NOT specify a message type whose vendor ID is 0xffffff or 
152 	* whose subtype is 0xff. These values are reserved for use as wild
153 	* cards, as described in section 3.9.1. If the IMC violates this 
154 	* prohibition, the TNCC SHOULD throw TNC_RESULT_INVALID_PARAMETER
155 	* exception.
156     * <p>
157     * @param messageFlags flags associated with the message
158     * @param messageVendorID vendor ID associated with the message
159     * @param messageSubtype message subtype associated with the message
160     * @param message the message to be delivered
161     * @param sourceIMCID source IMCID for the message
162     * @param destinationIMVID destination IMVID for the message
163     *
164     * @exception TNCException if a TNC error occurs
165     */
166     public void sendMessageLong(long messageFlags,
167 								long messageVendorID,
168 								long messageSubtype,
169 								byte[] message,
170 								long sourceIMCID,
171 								long destinationIMVID) throws TNCException;
172 }
