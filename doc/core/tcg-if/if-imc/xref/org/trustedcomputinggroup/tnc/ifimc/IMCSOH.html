52      /**
53  	 * The TNC Client calls this method (if supported by the TNCC and 
54  	 * implemented by an IMC) to deliver an SOHRReportEntry message to
55  	 * the IMC. This allows an IMC to receive the entire 
56  	 * SOHRReportEntry instead of just the contents of the 
57  	 * Vendor-Specific attribute, as would be the case if the 
58  	 * receiveMessage method was used.
59  	 * <p>
60  	 * A TNCC that supports IF-TNCCS-SOH SHOULD support this method 
61  	 * and a TNCC that supports this function SHOULD call this method 
62  	 * for a particular IMC and connection instead of calling 
63  	 * receiveMessage if the IMC implements this method and the 
64  	 * connection uses IF-TNCCS-SOH. A TNCC MUST NOT call this method 
65  	 * for a particular IMC and connection if the IMC does not 
66  	 * implement this function or the connection does not use 
67  	 * IF-TNCCS-SOH. A TNCC will often find that some IMCs loaded by 
68  	 * that TNCC implement this function and some do not. In that case,
69  	 * when the TNCC is handling a connection that uses IF-TNCCS-SOH, 
70  	 * the TNCC would call receiveMessageSOH for the IMCs that 
71  	 * implement this method and receiveMessage for the IMCs that do 
72  	 * not implement receiveMessageSOH. A TNCC MUST NOT call both 
73  	 * receiveMessage and receiveMessageSOH for a single 
74  	 * SOHRReportEntry for a single IMC.
75  	 * <p>
76  	 * IMCs are not required to implement this method but any IMC that
77  	 * implements the IMCSOH interface MUST implement this method. An 
78  	 * IMC should implement receiveMessageSOH if it wants to receive 
79  	 * the entire SOHRReportEntry instead of just the Vendor-Specific 
80  	 * attribute when IF-TNCCS-SOH is used. However, IMCs should 
81  	 * recognize that many TNCCs do not support IF-TNCCS-SOH and some 
82  	 * TNCCs that do support IF-TNCCS-SOH will not support this 
83  	 * function. Therefore, IMCs should be prepared to receive 
84  	 * messages via receiveMessage in some cases when IF-TNCC-SOH is 
85  	 * used. Simple IMCs that only send messages need not implement 
86  	 * this method. IMC implementers may find that implementing 
87  	 * receiveMessageSOH is more complex than implementing 
88  	 * receiveMessage since receiveMessageSOH must parse the 
89  	 * SOHRReportEntry while  receiveMessage only needs to parse the 
90  	 * contents of the Vendor-Specific attribute within the 
91  	 * SOHRReportEntry. For many IMCs, the contents of the 
92  	 * Vendor-Specific attribute is all that they need. Therefore, 
93  	 * IMCs are not required to implement receiveMessageSOH.
94  	 * <p>
95  	 * The contents of the SOHRReportEntry is contained in the buffer 
96  	 * referenced by sohrReportEntry. The length of this array MUST 
97  	 * NOT be zero and the parameter MUST NOT be null since zero-length
98  	 * SOHRReportEntries are not permitted by the IF-TNCCS-SOH 
99  	 * protocol. The contents of the first System-Health-ID attribute 
100 	 * in the SOHRReportEntry is indicated by systemHealthID. This 
101 	 * value MUST match one of the message type values previously 
102 	 * supplied by the IMC to the TNCC in the IMC's most recent call 
103 	 * to the TNCC's reportMessageTypes() or reportMessageTypesLong() 
104 	 * methods. IMCs MAY check these parameters to make sure they are 
105 	 * valid and throw an exception if not, but IMCs are not required 
106 	 * to make these checks. 
107 	 * <p>
108 	 * The IMC MUST NOT send any IMC-IMV messages on the connection 
109 	 * after this method is called since IF-TNCCS-SOH supports only 
110 	 * one round-trip.
111 	 * <p>
112 	 * As with all IMC methods, the IMC SHOULD NOT wait a long time 
113 	 * before returning from receiveMessageSOH. To do otherwise would 
114 	 * risk delaying the handshake indefinitely. A long delay might 
115 	 * frustrate users or exceed network timeouts (PDP, PEP or otherwise).
116 	 * <p>
117 	 * The IMC MUST NOT ever modify the buffer contents and MUST NOT 
118 	 * access the buffer after receiveMessageSOH has returned. If the 
119 	 * IMC wants to retain the contents of the SOHRReportEntry, it 
120 	 * should copy it before returning from receiveMessageSOH.
121 	 * <p>
122      * @param c the IMC connection object
123      * @param systemHealthID the type of message to be delivered
124      * @param sohrReportEntry the message to be delivered
125      *
126      * @exception TNCException if a TNC error occurs
127      */
128 	public void receiveMessageSOH(IMCConnection c,
129 								  long systemHealthID,
130 								  byte[] sohrReportEntry) throws TNCException;
131 }
