52     /**
53      * An IMC calls this method (if implemented by the TNCC) to give a 
54  	* SoHReportEntry to the TNCC for delivery. This allows an IMC to 
55  	* send an entire SoHReportEntry instead of just the contents of 
56  	* the Vendor-Specific attribute, as would be the case if the 
57  	* sendMessage method was used.
58      * <p>
59      * The SoHReportEntry (as defined in section 3.5 of [9]) is 
60  	* contained in the buffer referenced by the sohReportEntry 
61  	* parameter. If the length of the sohReportEntry parameter is 
62  	* zero (0), the TNCC MUST throw a TNCException with a 
63  	* TNC_RESULT_INVALID_PARAMETER result since a zero-length 
64  	* SoHReportEntry is prohibited by the IF_TNCCS-SOH 1.0 
65  	* specification. No type is included with the SoHReportEntry since
66  	* this is included in the System-Health-ID attribute in the 
67  	* SoHReportEntry. TNCCs MAY check these values to make sure they 
68  	* are valid and throw a TNCException with a 
69  	* TNC_RESULT_INVALID_PARAMETER result if not, but TNCCs are not 
70  	* required to make these checks.
71      * <p>
72      * A TNCC that supports IF-TNCCS-SOH SHOULD support this method. 
73  	* The TNC Client MUST NOT ever modify the buffer contents and MUST
74  	* NOT access the buffer after sendMessageSOH has returned. The TNC
75  	* Client will typically copy the SoHReportEntry out of the buffer,
76  	* place it into the SoH message or queue it up for such placement,
77  	* and return from this method.
78      * <p>
79      * An IMC may call this method if it needs to send an 
80  	* SoHReportEntry with a connection that supports IF-TNCCS-SOH (as 
81  	* indicated by the Has SOH attribute being set to a value of 1). 
82  	* If an IMC calls this method for a connection that does not 
83  	* support IF-TNCCS-SOH, the TNCC SHOULD throw a TNCException with 
84  	* a TNC_RESULT_NO_SOH_SUPPORT result. IMCs should recognize that 
85  	* many TNCCs do not support IF-TNCCS-SOH and some TNCCs that do 
86  	* support IF-TNCCS-SOH will not support this method. Therefore, 
87  	* IMCs should be prepared to send messages via sendMessage in some
88  	* cases when IF-TNCC-SOH is used. Simple IMCs that only need to 
89  	* send simple messages need not call this method. They can just 
90  	* call sendMessage, which will automatically place their message 
91  	* in a Vendor-Specific attribute in an SoHReportEntry.
92      * <p>
93      * The IMC MUST NOT call this method unless it has received a call 
94  	* to beginHandshake, receiveMessage, receiveMessageSOH, 
95  	* receiveMessageLong, or batchEnding for this connection and the 
96  	* IMC has not yet returned from that method. If the IMC violates 
97  	* this prohibition, the TNCC SHOULD throw 
98  	* TNC_RESULT_ILLEGAL_OPERATION exception. If an IMC really wants 
99  	* to communicate with an IMV at another time, it should call 
100 	* requestHandshakeRetry.
101     * <p>
102     * Note that a TNCC or TNCS MAY cut off IMC-IMV communications at 
103 	* any time for any reason, including limited support for long 
104 	* conversations in underlying protocols, user or administrator 
105 	* intervention, or policy. If this happens, the TNCC will throw a 
106 	* TNCException with a TNC_RESULT_ILLEGAL_OPERATION result.
107     * <p>
108     * If the TNCC supports limiting the message size or number of 
109 	* round trips and the limits are exceeded, the TNCC MUST throw a 
110 	* TNCException with a result of 
111 	* TNC_RESULT_EXCEEDED_MAX_MESSAGE_SIZE or 
112 	* TNC_RESULT_EXCEEDED_MAX_ROUND_TRIPS as the case may be.  An IMC 
113 	* can get the Maximum Message Size and Maximum Number of Round 
114 	* Trips by using the related Attribute ID within the getAttribute 
115 	* method.  The IMC SHOULD adapt its behavior to accommodate these 
116 	* limitations if available.
117     * <p>
118     * @param sohReportEntry sohReportEntry to be delivered
119     *
120     * @exception TNCException if a TNC error occurs
121     */
122     public void sendMessageSOH(byte[] sohReportEntry) throws TNCException;
123 }
