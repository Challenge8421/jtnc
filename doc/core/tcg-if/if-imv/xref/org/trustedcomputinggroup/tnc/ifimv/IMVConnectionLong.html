51  public interface IMVConnectionLong {
52  
53  	/**
54  	 * An IMV calls this method to give a message to the TNCS for 
55  	 * delivery. This method provides several features that sendMessage
56  	 * does not: longer 32 bit vendor ID and message subtype fields, 
57  	 * a messageFlags field, and the ability to specify a source IMV 
58  	 * ID and a desired or exclusive destination IMC ID.
59  	 * <p>
60  	 * The message is contained in the buffer referenced by message. 
61  	 * This parameter may be null to represent an empty message. The 
62  	 * type of the message is indicated by the messageVendorID and 
63  	 * messageSubtype parameters. TNCSs MAY check these values to make
64  	 * sure they are valid and throw an exception if not, but TNCSs 
65  	 * are not required to make these checks.
66  	 * <p>
67  	 * The messageFlags parameter supports up to 32 flags that may be
68  	 * set pertaining to the message. At this time, only one flag has 
69  	 * been defined. The value 0x80000000 is known as the
70  	 * TNC_MESSAGE_FLAGS_EXCLUSIVE flag ("the EXCL flag", for short). 
71  	 * If this bit is set, the sending IMV is requesting that the 
72  	 * message only be delivered to the IMC designated by the 
73  	 * destinationIMCID parameter. If the bit is cleared, the message 
74  	 * is to be delivered to any IMC that has indicated an interest in
75  	 * the message type. Other flags may be defined in future versions
76  	 * of this specification. Until that time, IMVs MUST leave all 
77  	 * these flags cleared. Some connections and/or TNCSs may not 
78  	 * support the EXCL flag. To determine whether a particular TNCS 
79  	 * and connection supports this flag, the IMV should get the Has 
80  	 * Exclusive attribute for that connection. If the IMV sets a flag
81  	 * that is not supported by the TNCS or the connection, the TNCS 
82  	 * SHOULD throw a TNCException with a TNC_RESULT_INVALID_PARAMETER
83  	 * result.
84  	 * <p>
85  	 * The IMV MUST set the sourceIMVID parameter to either the primary
86  	 * IMV ID queried by IMV using the TNC_AttributeID_PRIMARY_IMV_ID
87  	 * attribute or an additional IMV ID reserved when IMV requests 
88  	 * the TNCS to do so by calling reserveAdditionalIMVID. This IMV ID
89  	 * will be used as the sourceIMVID for protocols that support 
90  	 * sending this field along with the message.
91  	 * <p>
92  	 * If the EXCL flag is set, the IMV MUST set the destinationIMCID 
93  	 * parameter to indicate which IMC should receive the message being
94  	 * sent. The IMV should place into this parameter a value which it
95  	 * has obtained using the sourceIMCID parameter for messages 
96  	 * previously delivered to the IMV on this connection. The IMV MAY
97  	 * set the destinationIMCID parameter to a specific IMC ID even 
98  	 * when the EXCL flag is not set. This indicates that the message 
99  	 * being sent is in response to a message received from the IMC 
100 	 * indicated in the destinationIMCID parameter. If the IMV does 
101 	 * not know the IMC ID of the recipient IMC, the IMV MUST use the 
102 	 * TNC_IMCID_ANY wild card for the destinationIMCID parameter and 
103 	 * MUST NOT set the EXCL flag. In such case, the message is 
104 	 * delivered to all IMCs that have expressed interest in receiving
105 	 * such messages. This may happen when the TNCS starts the 
106 	 * integrity handshake and the IMV does not know the IMC IDs of 
107 	 * the recipient IMCs.
108 	 * <p>
109 	 * TNCSs are not required to implement this method but any TNCS 
110 	 * that implements the IMVConnectionLong interface MUST implement 
111 	 * this method. Since this method was not included in IF-IMV 1.2, 
112 	 * many TNCSs do not implement it. IMVs MUST work properly if a 
113 	 * TNCS does not implement this method. The IMV is never required 
114 	 * to call this method. The TNCS MUST work with IMVs that don't 
115 	 * call this method. The IMV also SHOULD check the Has Long 
116 	 * attribute for a given connection to determine whether this 
117 	 * connection supports long types before calling this method 
118 	 * for that connection. If the connection does not support long 
119 	 * types, the IMV MAY call this method (if implemented) but MUST 
120 	 * NOT use a messageVendorID greater than 0xffffff or a 
121 	 * messageSubtype greater than 0xff.  If the IMV does so, the TNCS
122 	 * SHOULD throw a TNC_RESULT_NO_LONG_TYPES exception.
123 	 * <p>
124 	 * The TNC Server MUST NOT ever modify the buffer contents and 
125 	 * MUST NOT access the buffer after sendMessageLong has returned. 
126 	 * The TNC Server will typically copy the message out of the 
127 	 * buffer, queue it up for delivery, and return from this method.
128 	 * <p>
129 	 * The IMV MUST NOT call this method unless it has received a call
130 	 * to beginHandshake, receiveMessage, receiveMessageSOH, 
131 	 * receiveMessageLong, or batchEnding for this connection and the 
132 	 * IMV has not yet returned from that method. If the IMV violates 
133 	 * this prohibition, the TNCS SHOULD throw 
134 	 * TNC_RESULT_ILLEGAL_OPERATION exception. If an IMV really wants 
135 	 * to communicate with an IMC at another time, it should call 
136 	 * requestHandshakeRetry.
137 	 * <p>
138 	 * Note that a TNCC or TNCS MAY cut off IMC-IMV communications at
139 	 * any time for any reason, including limited support for long 
140 	 * conversations in underlying protocols, user or administrator 
141 	 * intervention, or policy. If this happens, the TNCS will throw
142 	 * TNC_RESULT_ILLEGAL_OPERATION exception from sendMessageLong. 
143 	 * If the TNCS supports limiting the message size or number of 
144 	 * round trips, the TNCS MUST throw 
145 	 * TNC_RESULT_EXCEEDED_MAX_MESSAGE_SIZE or 
146 	 * TNC_RESULT_EXCEEDED_MAX_ROUND_TRIPS exception respectively if 
147 	 * the limits are exceeded.  An IMV can get the Maximum Message 
148 	 * Size and Maximum Number of Round Trips by using the related 
149 	 * Attribute ID within the getAttribute method.  The IMV SHOULD 
150 	 * adapt its behavior to accommodate these limitations if available.
151 	 * <p>
152 	 * The TNC Server MUST support any message type. However, the IMV 
153 	 * MUST NOT specify a message type whose vendor ID is 0xffffff or 
154 	 * whose subtype is 0xff. These values are reserved for use as 
155 	 * wild cards, as described in section 3.9.1. If the IMV violates 
156 	 * this prohibition, the TNCS SHOULD throw 
157 	 * TNC_RESULT_INVALID_PARAMETER exception.
158 	 * <p>
159 	 * @param messageFlags flags associated with the message
160 	 * @param messageVendorID vendor ID associated with the message
161 	 * @param messageSubtype message subtype associated with the message
162 	 * @param message the message to be delivered
163 	 * @param sourceIMVID source IMC ID for the message
164 	 * @param destinationIMCID destination IMV ID for message
165 	 *
166 	 * @throws TNCException if a TNC error occurs
167 	 */
168 	public void sendMessageLong(long messageFlags,
169 							    long messageVendorID,
170 							    long messageSubtype,
171 							    byte[] message,
172 							    long sourceIMVID,
173 							    long destinationIMCID) throws TNCException;
174 }
