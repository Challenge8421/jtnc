50  public interface IMVSOH {
51  
52  	/**
53  	 * The TNC Server calls this method (if supported by the TNCS and 
54  	 * implemented by an IMV) to deliver an SOHReportEntry message to 
55  	 * the IMV. This allows an IMV to receive the entire SOHReportEntry
56  	 * instead of just the contents of the Vendor-Specific attribute, 
57  	 * as would be the case if the receiveMessage method was used.
58  	 * <p>
59  	 * A TNCS that supports IF-TNCCS-SOH SHOULD support this method 
60  	 * and a TNCS that supports this function SHOULD call this method 
61  	 * for a particular IMV and connection instead of calling 
62  	 * receiveMessage if the IMV implements this method and the 
63  	 * connection uses IF-TNCCS-SOH. A TNCS MUST NOT call this method 
64  	 * for a particular IMV and connection if the IMV does not implement
65  	 * this function or the connection does not use IF-TNCCS-SOH. A 
66  	 * TNCS will often find that some IMVs loaded by that TNCS implement
67  	 * this function and some do not. In that case, when the TNCS is 
68  	 * handling a connection that uses IF-TNCCS-SOH, the TNCS would 
69  	 * call receiveMessageSOH for the IMVs that implement this method 
70  	 * and receiveMessage for the IMVs that do not implement 
71  	 * receiveMessageSOH. A TNCS MUST NOT call both receiveMessage and
72  	 * receiveMessageSOH for a single SOHReportEntry for a single IMV.
73  	 * <p>
74  	 * IMVs are not required to implement this method but any IMV that 
75  	 * implements the IMVSOH interface MUST implement this method. An 
76  	 * IMV should implement receiveMessageSOH if it wants to receive 
77  	 * the entire SOHReportEntry instead of just the Vendor-Specific 
78  	 * attribute when IF-TNCCS-SOH is used. However, IMVs should 
79  	 * recognize that many TNCSs do not support IF-TNCCS-SOH and some 
80  	 * TNCSs that do support IF-TNCCS-SOH will not support this 
81  	 * function. Therefore, IMVs should be prepared to receive messages
82  	 * via receiveMessage in some cases when IF-TNCC-SOH is used. 
83  	 * Simple IMVs that do not wish to receive the entire SOHReportEntry 
84  	 * need not implement this function. IMV implementers may find that 
85  	 * implementing receiveMessageSOH is more complex than implementing 
86  	 * receiveMessage since receiveMessageSOH must parse the 
87  	 * SOHReportEntrywhile  receiveMessage only needs to parse the 
88  	 * contents of the Vendor-Specific attribute within the 
89  	 * SOHReportEntry. For many IMVs, the contents of the 
90  	 * Vendor-Specific attribute is all that they need. Therefore, 
91  	 * IMVs are never required to implement receiveMessageSOH.
92  	 * <p>
93  	 * The contents of the SOHReportEntry is contained in the buffer 
94  	 * referenced by sohReportEntry. This parameter may be null to 
95  	 * represent an empty message. The contents of the first 
96  	 * System-Health-ID attribute in the SOHReportEntry is indicated 
97  	 * by systemHealthID. This value MUST match one of the message 
98  	 * type values previously supplied by the IMV to the TNCS in the 
99  	 * IMV's most recent call to the TNCS's reportMessageTypes() or 
100 	 * reportMessageTypesLong() methods. IMVs MAY check these parameters
101 	 * to make sure they are valid and throw an exception if not, but 
102 	 * IMVs are not required to make these checks.
103 	 * <p>
104 	 * The IMV SHOULD send any IMC-IMV messages it wants to send as 
105 	 * soon as possible after this method is called and then return 
106 	 * from this method to indicate that it is finished sending 
107 	 * messages in response to this message.
108 	 * <p>
109 	 * As with all IMV methods, the IMV SHOULD NOT wait a long time 
110 	 * before returning from receiveMessageSOH. To do otherwise would 
111 	 * risk delaying the handshake indefinitely. A long delay might 
112 	 * frustrate users or exceed network timeouts (PDP, PEP or 
113 	 * otherwise).
114 	 * <p>
115 	 * The IMV MUST NOT ever modify the buffer contents and MUST NOT 
116 	 * access the buffer after receiveMessageSOH has returned. If the 
117 	 * IMV wants to retain the contents of the SOHReportEntry, it 
118 	 * should copy it before returning from receiveMessageSOH.
119 	 * <p>
120 	 * @param c the IMV connection object
121 	 * @param systemHealthID the type of message to be delivered
122 	 * @param sohReportEntry the message to be delivered
123 	 *
124 	 * @throws TNCException if a TNC error occurs
125 	 */
126 	void receiveMessageSOH(IMVConnection c,
127 						   long systemHealthID,
128 						   byte[] sohReportEntry) throws TNCException;
129 }
