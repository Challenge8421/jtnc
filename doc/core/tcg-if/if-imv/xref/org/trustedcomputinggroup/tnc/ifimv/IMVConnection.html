3   /**
4    * The IMV and TNCS use this IMVConnection object to 
5    * refer to the network connection when delivering 
6    * messages and performing other operations relevant to 
7    * the network connection. This helps ensure that IMV 
8    * messages are sent to the right TNCC and IMCs, helps 
9    * ensure that the IMV Action Recommendation is 
10   * associated with the right endpoint, and helps the IMV 
11   * match up messages from IMCs with any state the IMV may 
12   * be maintaining from earlier parts of that IMC-IMV 
13   * conversation (even extending across multiple 
14   * Integrity Check Handshakes in a single network 
15   * connection).
16   * <p>
17   * The TNCS MUST create a new IMVConnection object for each
18   * combination of an IMV and a connection. IMVConnection
19   * objects MUST NOT be shared between multiple IMVs.
20   */

147     /**
148      * Gives a message to the TNCS for
149      * delivery. The message is contained in the buffer referenced by 
150      * the <code>message</code> parameter.  The <code>message</code> parameter may 
151      * be null which represent an empty message. The type 
152      * of the message is indicated by the messageType parameter. 
153      * <p>
154      * All IMVConnections MUST implement this method. An
155      * IMVConnection MUST NOT ever modify the buffer contents and 
156      * MUST NOT access the buffer after the sendMessage
157      * method has returned. The IMVConnection will 
158      * typically copy the message out of the buffer, queue it up 
159      * for delivery, and return from this method.
160      * <p>
161      * The IMV MUST NOT call this method unless it has 
162      * received a call to the IMV's receiveMessage method or the 
163      * IMV's batchEnding method for this connection and the IMV 
164      * has not yet returned from that method. If the IMV 
165      * violates this prohibition, the TNCS SHOULD throw the 
166      * TNC_RESULT_ILLEGAL_OPERATION exception. If an IMV 
167      * really wants to communicate with an IMC at another 
168      * time, it should call the IMVConnection's requestHandshakeRetry
169      * method.
170      * <p>
171      * Note that a TNCC or TNCS MAY cut off IMC-IMV 
172      * communications at any time for any reason, including 
173      * limited support for long conversations in underlying 
174      * protocols, user or administrator intervention, or 
175      * policy. If this happens, the IMVConnection's
176      * sendMessage method will 
177      * throw a TNCException with result code TNC_RESULT_ILLEGAL_OPERATION
178      * and call the IMVs'
179      * solicitRecommendation() to elicit IMV Action 
180      * Recommendations based on the data they have gathered 
181      * so far. If the TNCS supports limiting the message size 
182 	 * or number of round trips, the TNCS MUST throw a TNCException
183 	 * with result code TNC_RESULT_EXCEEDED_MAX_MESSAGE_SIZE 
184 	 * or TNC_RESULT_EXCEEDED_MAX_ROUND_TRIPS respectively if 
185 	 * the limits are exceeded.  An IMV can get the Maximum 
186 	 * Message Size and Maximum Number of Round Trips by using 
187 	 * the related Attribute ID with the getAttribute method on 
188 	 * the TNCS object.  The IMV SHOULD adapt its behavior to 
189 	 * accommodate these limitations if available.
190      * <p>
191      * The TNC Server MUST support any message type. However, 
192      * the IMV MUST NOT specify a message type whose vendor 
193      * ID is 0xffffff or whose subtype is 0xff. These values 
194      * are reserved for use as wild cards, as described in 
195      * the Abstract API. If the IMV violates this prohibition, 
196      * the IMVConnection SHOULD throw a TNCException
197      * with result code TNC_RESULT_INVALID_PARAMETER. If the 
198 	 * IMVConnection supports limiting the message size or number
199 	 * of round trips, the IMVConnection MUST return 
200 	 * TNC_RESULT_EXCEEDED_MAX_MESSAGE_SIZE exception or 
201 	 * TNC_RESULT_EXCEEDED_MAX_ROUND_TRIPS exception respectively
202 	 * if the limits are exceeded.  An IMV can get the Maximum 
203 	 * Message Size and Maximum Number of Round Trips by using
204 	 * the related TNC_ATTRIBUTEID within the TNCS.getAttribute
205 	 * method.  The IMV SHOULD adapt its behavior to accommodate
206 	 * these limitations if available.
207      *
208      * @param messageType the type of message to be delivered
209      * @param message the message to be delivered 
210      *
211      * @throws TNCException if an error occurs
212      */
213     
214     public void sendMessage(long messageType, byte[] message) throws TNCException;
215     
216     /**
217      * Asks a TNCS to retry an Integrity Check Handshake for this
218      * IMVConnection. The IMV MUST pass one of the 
219      * handshake retry reasons listed in the Abstract API as the 
220      * reason parameter. 
221      * <p>
222      * TNCSs MAY check the parameters to make sure they are 
223      * valid and throw an exception if not, but TNCSs are not 
224      * required to make these checks. The reason parameter 
225      * explains why the IMV is requesting a handshake retry. 
226      * The TNCS MAY use this in deciding whether to attempt 
227      * the handshake retry. As noted in the Abstract API, TNCSs 
228      * are not required to honor IMV requests for handshake 
229      * retry (especially since handshake retry may not be 
230      * possible or may interrupt network connectivity). An 
231      * IMV MAY call this method at any time, even if an 
232      * Integrity Check Handshake is currently underway. 
233      * This is useful if the IMV suddenly gets important 
234      * information but has already finished its dialog with 
235      * the IMC, for instance. As always, the TNCS is not 
236      * required to honor the request for handshake retry.
237      * <p>
238      * If the TNCS cannot attempt the handshake retry, 
239      * the IMVConnection SHOULD throw a TNCException
240      * with result code TNC_RESULT_CANT_RETRY.
241      * If the TNCS could attempt to retry the handshake but 
242      * chooses not to, the IMVConnection SHOULD throw a TNCException
243      * with result code TNC_RESULT_WONT_RETRY.
244      * The IMV MAY use this information 
245      * in displaying diagnostic and progress messages.
246      * <p>
247      * @param reason handshake retry reason code
248      *
249      * @throws TNCException
250      */
251     public void requestHandshakeRetry(long reason) throws TNCException;
252     
253     /**
254      * An IMV calls this method to deliver its IMV 
255      * Action Recommendation and IMV Evaluation Result to 
256      * the TNCS. The TNCS SHOULD use the recommendation 
257      * value in determining its own TNCS Action 
258      * Recommendation or decision about endpoint access. 
259      * The TNC specifications do not specify how the TNCS 
260      * does the recommendation value but it is certainly 
261      * essential to have a recommendation from the IMV. 
262      * The TNC specifications also do not specify what 
263      * the TNCS does with the evaluation value. It may 
264      * log it.
265      * <p>
266      * The IMV MUST pass one of the IMV Action 
267      * Recommendation values listed in the Abstract API
268      * as the recommendation parameter and one of 
269      * the IMV Evaluation Result values listed in the Abstract 
270      * API as the evaluation parameter. TNCSs 
271      * MAY check these values to make sure they are 
272      * valid and throw an exception if not, but TNCSs are 
273      * not required to make these checks.
274      * <p>
275      * The IMV should deliver its IMV Action 
276      * Recommendation as soon as possible so that the 
277      * TNCS can proceed with determining its own TNCS 
278      * Action Recommendation. If the IMV receives a 
279      * message from an IMC and is able to decide on 
280      * an IMV Action Recommendation and deliver it 
281      * to the TNCS before returning from the IMV 
282      * receiveMessage method, it SHOULD do so. 
283      * However, as always the IMV SHOULD return 
284      * promptly to avoid a long delay that might 
285      * frustrate users or exceed network timeouts (PDP, 
286      * PEP or otherwise).
287      * <p>
288      * An IMV SHOULD NOT expect that it will be able 
289      * to send IMC-IMV messages after calling the 
290      * IMVConnection's provideRecommendation method. The TNCS 
291      * may decide to terminate the handshake immediately 
292      * based on the IMV Action Recommendation. For 
293      * instance, IMVs SHOULD send remediation 
294      * instructions before calling the IMVConnection's
295      * provideRecommendation method.
296      * <p>
297      * However, a TNCS MAY continue to deliver messages 
298      * after an IMV calls the IMVConnection's provideRecommendation method, 
299      * especially if other IMVs continue the dialog after
300      * the one IMV has rendered its decision. The IMV 
301      * MUST be prepared for this. It MAY simply ignore 
302      * these late messages or it MAY consider them and 
303      * even change its recommendation by calling the IMVConnection's
304      * provideRecommendation method again. In this 
305      * case, the TNCS SHOULD use the last recommendation 
306      * received from an IMV during a particular handshake. 
307      * However, the TNCS is not required to do this.
308      * <p>
309      * If an IMV does not provide a recommendation 
310      * earlier, the TNCS will call the IMV's
311      * solicitRecommendation method at the end of an 
312      * Integrity Check Handshake (after all IMC-IMV 
313      * messages have been delivered). The IMV SHOULD 
314      * then call the IMVConnection's provideRecommendation method to 
315      * deliver its recommendation. If the IMV calls 
316      * this method when there is no active handshake 
317      * on the specified network connection, the TNCS 
318      * SHOULD throw the TNC_RESULT_ILLEGAL_OPERATION 
319      * exception. If an IMV really needs to communicate 
320      * a recommendation at another time, it should call
321      * the IMVConnection's requestHandshakeRetry method.
322      * <p>
323      * @param recommendation action recommendation
324      * @param evaluation evaluation result
325      *
326      * @throws TNCException
327      */
328     
329     public void provideRecommendation(long recommendation, long evaluation) throws TNCException;

331     /**
332      * An IMV calls this method to get the value of the attribute
333      * identified by <code>attributeID</code> for this
334      * <code>IMVConnection</code>.
335      * <p>
336      * This function is optional. The TNCS is not required to implement
337      * it. If it is not implemented for this IMVConnection, it MUST
338      * throw an UnsupportedOperationException. IMVs MUST work properly
339      * if a TNCS does not implement this function.
340      * <p>
341      * The IMV MUST pass a standard or vendor-specific attribute ID as
342      * the attributeID parameter. If the TNCS does not recognize the
343      * attribute ID, it SHOULD throw a TNCException with the
344      * TNC_RESULT_INVALID_PARAMETER result code. If the TNCS
345      * recognizes the attribute ID but does not have an attribute
346      * value for the requested attribute ID for this
347      * <code>IMVConnection</code>, it SHOULD also throw a TNCException
348      * with the TNC_RESULT_INVALID_PARAMETER result code.
349      * <p>
350      * The return value is an <code>Object</code> that represents the
351      * attribute value requested. The IMV must cast this <code>Object</code>
352      * to the class documented in the description of that specific
353      * attribute to get the desired value. All <code>Object</code>s
354      * returned by this method SHOULD be immutable.
355      *
356      * @param attributeID the attribute ID of the desired attribute
357      * @return the attribute value
358      * @throws TNCException
359      */
360     
361     public Object getAttribute(long attributeID) throws TNCException;
362     
363     /**
364      * An IMV calls this method to set the value of the attribute
365      * identified by <code>attributeID</code> for this
366      * <code>IMVConnection</code>.
367      * <p>
368      * This function is optional. The TNCS is not required to implement
369      * it. If it is not implemented for this IMVConnection, it MUST
370      * throw an UnsupportedOperationException. IMVs MUST work properly
371      * if a TNCS does not implement this function.
372      * <p>
373      * The IMV MUST pass a standard or vendor-specific attribute ID as
374      * the attributeID parameter. If the TNCS does not recognize the
375      * attribute ID, it SHOULD throw a TNCException with the
376      * TNC_RESULT_INVALID_PARAMETER result code. If the TNCS
377      * recognizes the attribute ID but does not support setting
378      * an attribute value for the requested attribute ID for this
379      * <code>IMVConnection</code>, it SHOULD also throw a TNCException
380      * with the TNC_RESULT_INVALID_PARAMETER result code.
381      * <p>
382      * For the <code>attributeValue</code> parameter, the IMV MUST pass
383      * an <code>Object</code> that represents the new attribute value
384      * (or <code>null</code> if permitted for the specified attribute).
385      * This <code>Object</code> must actually be an instance of the
386      * class documented in the description of the specified attribute.
387      * The <code>Object</code> SHOULD be immutable. If the TNCS has
388      * any uncertainty about it SHOULD copy the object. The TNCS MAY
389      * check the <code>Object</code> and throw a <code>TNCException</code>
390      * if it is not a valid value for the specified attribute.
391      *
392      * @param attributeID the attribute ID of the attribute to be set
393      * @param attributeValue the new value to be set for this attribute
394      * @throws TNCException
395      */
396     public void setAttribute(long attributeID, Object attributeValue)
397 	throws TNCException;
